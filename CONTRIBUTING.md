# 🤝 기여 가이드 (Contributing Guidelines)

본 프로젝트는 4인 개발 체제(TA 1명, UE 프로그래머 2명, AI 엔지니어 1명)로 단기간(8월 26일 최종 발표) 내에 완성해야 하는 오픈월드 아포칼립스 생존 게임입니다. 파일 충돌을 최소화하고 효율성을 극대화하기 위해 아래 규칙을 엄격히 준수합니다.

## 👥 직무별 작업 영역 및 규칙 (Role Guidelines)

### 🎨 Technical Artist (TA - 1명)
- **작업 폴더:** `Content/Art/`, `Content/TechArt/`
- **에셋 적용:** 원본 작업 파일(`.psd`, `.blend`, `.spp` 등)은 용량 문제로 Git에 직접 올리지 않고 공용 드라이브에 보관합니다. Git에는 언리얼 엔진용으로 임포트된 에셋(`.uasset`)만 커밋합니다.
- **협업:** 마스터 머티리얼 등 공용 리소스를 수정할 때는 다른 팀원의 맵이 깨지지 않도록 사전에 공유합니다.

### 💻 UE Programmers (2명)
- **작업 폴더:** `Source/`, `Content/Blueprints/`
- **구조 원칙 (Component-Based):** 
  - 코어 로직(상태 관리, 인벤토리 등)은 C++ 액터 컴포넌트로 작성하여 독립적으로 작동하게 합니다.
  - 기획 및 TA가 수치를 조절할 수 있도록 Data Asset을 적극 활용합니다.
  - 두 명의 프로그래머가 서로 다른 컴포넌트 시스템을 담당하여 소스 코드 충돌을 예방합니다.

### 🤖 AI Engineer (1명)
- **작업 폴더:** `AI/`, `Source/Companion/`
- **구조 원칙:** AI 동료의 판단 로직(LLM)과 언리얼 엔진(행동 트리 및 이동) 간의 책임을 명확히 분리합니다.
- **⚠️ 보안 및 환경설정 (.env) 규칙:**
  - LLM API Key 및 민감한 설정값은 절대 Git에 커밋하지 않습니다.
  - 반드시 로컬 `AI/` 폴더 내에 `.env` 파일을 생성하여 키를 관리하세요. (팀원 공유용으로 키 값이 비어있는 `.env.example` 파일을 깃에 올리는 것을 권장합니다.)
  - **[중요]** 새로운 API 키를 발급받거나 데이터셋을 추가한 경우, **커밋(Commit)하기 전에 반드시 `git status`를 확인하세요.** `.env` 파일이나 데이터셋 폴더가 추적(tracked) 목록에 뜨지 않는지 확인하여, `.gitignore`가 정상적으로 작동하고 있는지 2차 검수해야 합니다.

## 🌿 브랜치 전략 (Branch Strategy)
- `main` : 마일스톤 평가를 위한 안정적인 배포 브랜치
- `develop` : 개발용 메인 브랜치 (모든 PR의 타겟 브랜치)
- `feature/[Role]/[Issue-Number]-[Task]` : 개별 기능 브랜치 (예: `feature/UE/#12-inventory-ui`)
- `bugfix/[Role]/[Issue-Number]-[Task]` : 버그 수정 브랜치 (예: `bugfix/AI/#15-llm-crash`)

## 📝 커밋 메시지 규칙 (Commit Convention)
협업 시 히스토리 파악과 이슈 추적을 용이하게 하기 위해, 아래 포맷을 엄격하게 지켜주세요.

### 1. 커밋 메시지 구조
```text
[타입] 제목 (50자 이내, 명확하게 작성)

본문 (선택 사항, 어떻게/왜 변경했는지 구체적인 사유나 내용 작성)

해결한 이슈 번호 (선택 사항)
```

### 2. 타입 (Types)
- `[Feat]` : 새로운 기능 구현, 블루프린트 시스템 추가, 레벨 생성
- `[Fix]` : 버그, 크래시, 치명적인 오류 수정
- `[Design]` : 레벨 디자인 배치, UI 레이아웃 변경, 머티리얼 및 시각 효과 수정
- `[Asset]` : 외부 3D 모델, 애니메이션, 사운드 등 신규 에셋 Import
- `[Docs]` : 리드미, 기여 가이드, 코드 내 주석 등 문서화 작업
- `[Refactor]` : 코드 구조 및 설계 개선 (외부 기능 변경 없음)
- `[Chore]` : 언리얼 빌드 설정, `.gitignore` 수정, 패키징 관련 자잘한 변경

### 3. 올바른 커밋 예시
```text
[Feat] 플레이어 허기 및 갈증 생존 컴포넌트 추가

- UPlayerSurvivalComponent를 생성하여 시간에 따른 체력 감소 로직 구현
- 기획/TA가 수치를 조절할 수 있도록 SurvivalDataAsset 연동
- UI에 스탯 반영할 수 있게 델리게이트 이벤트 추가

Resolves: #24
```

## ⚠️ 병합(Merge) 및 에셋 충돌 방지
1. **블루프린트 & 레벨 에셋 독점:** 언리얼 바이너리 에셋(`.uasset`, `.umap`)은 코드가 아니므로 병합(Merge)이 불가능합니다. 메인 블루프린트 수정 전 반드시 팀 메신저에 "OOO 블루프린트 작업 시작합니다"라고 알리고 **독점권(Lock)**을 가집니다.
2. **독립된 테스트 맵 사용:** 각자 자신만의 샌드박스 레벨(`Content/Maps/Test/[Name]_TestMap.umap`)에서 새로운 기능을 개발 및 테스트한 후, 문제가 없을 때 메인 레벨에 통합합니다.
